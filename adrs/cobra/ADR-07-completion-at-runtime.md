# ADR-07: Shell completions are produced at TAB time by executing the CLI binary with a hidden `__complete` subcommand

## Context

Shell tab-completion for a CLI has two extreme designs:

- **Static scripts.** The CLI ships a hand-written or generated bash/
  zsh/fish/powershell script that encodes every command, flag, and
  valid value. The shell evaluates that script on every TAB. The
  completion logic is in the script, expressed in whatever scripting
  language that shell supports. Typical examples: `git`'s bash-
  completion, `docker`'s packaged scripts.
- **Dynamic dispatch.** The shell forwards TAB requests back to the
  CLI binary itself via a small stub. The CLI computes the
  completions using its own code (including arbitrary Go logic —
  hitting an API, reading a file, computing a schema) and returns
  the list to the shell. Examples: `kubectl`, `gh`, `docker` (more
  recent), most modern CLIs.

The static approach is simple but limits completions to what a
shell script can compute — which is typically string constants,
glob patterns, and calls to other shell programs. Dynamic
completion of "the actual S3 buckets this user has access to" is
not viable in a bash script. Completion of "a flag's value in a
way that knows about already-set flags in the current invocation"
is extremely hard.

The dynamic approach is more powerful but requires a protocol:
the CLI must have some entry point the shell knows to call, that
entry point must understand where in the command line the user's
cursor is, and the return format must be something the shell can
parse. Each shell has its own conventions — bash uses `COMPREPLY`,
zsh uses `_describe`, fish uses `complete -C`, PowerShell uses
`Register-ArgumentCompleter`.

Cobra commits hard to dynamic completion. The library ships a
single Go implementation of the completion engine, invoked across
all four shells via thin shell-script adapters. The shell adapter
generation is the only shell-specific code; the completion logic
itself is Go.

The architectural implication is significant: **every binary built
with cobra is both an application and a completion engine.** When
a user TABs, their CLI runs.

## Decision

Cobra installs a hidden subcommand `__complete` (and
`__completeNoDesc`) on every root command. The shell completion
scripts generated by cobra invoke these. The generated script is
small; the logic lives in Go.

**The hidden completion commands are registered at execute time.**
`initCompleteCmd()` in `completions.go` (around lines 231-296)
creates the subcommand with `Hidden: true` and `DisableFlagParsing:
true`. It takes raw args (the user's partially-typed command line),
replays `Find()` against the rest of the command tree, and produces
completions by:

1. Resolving which command the user is completing against.
2. Calling that command's `ValidArgsFunction` or iterating its
   `ValidArgs` to get possible positional values.
3. Calling any `RegisterFlagCompletionFunc`-registered functions
   for flag values.
4. Consulting annotations for filename-completion or custom
   completion hints.
5. Formatting the result with the right `ShellCompDirective`.

`ShellCompDirective` is a bitset (in `completions.go`) with flags
like `ShellCompDirectiveNoSpace`, `ShellCompDirectiveNoFileComp`,
`ShellCompDirectiveFilterFileExt`, `ShellCompDirectiveKeepOrder`,
`ShellCompDirectiveError`. The completion function returns
`(completions []string, directive ShellCompDirective)`. This lets
Go code tell the shell "these are the candidates and also don't
add a trailing space" — which is critical for multi-stage
completion (flag value → another flag).

**Shell adapters are generated, per-shell, from templates.**
- `bash_completions.go` (706 lines) — V1 bash generator, older
  format, eval-heavy.
- `bash_completionsV2.go` (485 lines) — V2 bash generator, cleaner
  layering, supports descriptions and ActiveHelp.
- `zsh_completions.go` — generates a zsh completion script that
  calls `__complete`.
- `fish_completions.go` — fish variant.
- `powershell_completions.go` — PowerShell variant.

Each generator emits a script whose main job is: (a) wire the shell's
completion hook for the CLI's name, (b) when that hook fires, build
a command line that invokes `$binary __complete <args>`, (c) parse
the output and feed it to the shell's completion machinery.

**V1 and V2 bash coexist.** V2 was added because V1 had fundamental
limits:

```go
// bash_completionsV2.go disables ActiveHelp for V1:
// "Disable ActiveHelp which is not supported for bash completion v1"
```

V1 cannot render completion descriptions (fish and zsh natively do),
cannot support `ShellCompDirectiveKeepOrder` correctly, and cannot
surface ActiveHelp messages. V2 fixes these but uses bash 4+ features
and slightly different shell idioms. V1 stays for bash-3-and-earlier
environments and for back-compat with existing sourced scripts.

**The completion API surface for CLI authors:**

1. **`ValidArgs []Completion`** (`command.go:85-86`) — static list of
   positional values. `Completion` carries a name and optional
   description.
2. **`ValidArgsFunction CompletionFunc`** (`command.go:87-90`) —
   dynamic function that returns completions at TAB time. Cannot be
   used together with `ValidArgs`.
3. **`RegisterFlagCompletionFunc(flagName, func)`** — attach a
   completion function to a specific flag value.
4. **Annotations** — `BashCompFilenameExt` (complete filenames with
   certain extensions), `BashCompSubdirsInDir` (complete subdirs),
   `BashCompOneRequiredFlag` (marker for required flags).
5. **`ActiveHelp` messages** — dynamic help text shown inline during
   completion via `cobra.AppendActiveHelp(comps, msg)` in
   `active_help.go`. The shell adapter renders them as informational
   lines rather than candidates.

**`CompletionOptions` struct** (`command.go:207-208`). Controls
whether cobra adds a `completion` subcommand at all, whether
descriptions are shown, whether `__complete` is hidden, etc.
Programs that don't want cobra's built-in completion setup can
disable it here.

**Generation commands.** `GenBashCompletion(w)`,
`GenBashCompletionV2(w, includeDesc)`, `GenZshCompletion(w)`,
`GenFishCompletion(w, includeDesc)`, `GenPowerShellCompletion(w)`.
These emit the shell script to a writer. The built-in
`completion` subcommand (auto-added unless disabled) exposes
these as `myapp completion bash > /etc/bash_completion.d/myapp`
for end-user installation.

**The rule:** shell completion is a two-part system — a small
generated shell script that knows how to talk to its specific
shell, and a large Go completion engine embedded in the CLI
binary itself. The shell script never computes completions; it
always delegates to `__complete`. New shells are supported by
adding new generators, not by duplicating completion logic.

## Rationale

- **Dynamic completion is the whole point.** The motivating use
  cases — "tab-complete the cluster names this user has access
  to," "complete Kubernetes pod names in the current namespace,"
  "complete branch names from the configured remote" — require
  running the CLI's own code. Static scripts cannot do this.
  Cobra's architectural commitment is to the dynamic model; the
  static-script format is not on the roadmap.

- **One completion engine in Go, N shell adapters.** Every shell
  has unique completion conventions, but the *logic* of
  completion (walk tree, identify cursor position, call completion
  functions, format result) is shell-agnostic. Putting the logic
  in Go means writing it once. Each shell gets a small adapter
  script that is mostly "parse args, fork the binary, format
  result." This is the standard pattern for modern multi-shell
  CLI completion and it is the reason cobra can support four
  shells without four parallel implementations.

- **Running the CLI at TAB time is acceptable.** Launching a Go
  binary on TAB costs a few milliseconds — well within the
  threshold where users don't notice latency. For completion
  functions that hit network resources, the latency is dominated
  by the network call, which is unavoidable. Cobra's
  responsibility is to make the local overhead small, which it
  does by having `__complete` skip most of cobra's normal
  startup (flag inheritance runs but none of the Pre/Run/Post
  pipeline).

- **`ShellCompDirective` is a bitset because completion is
  multi-dimensional.** A completion result is "list of candidates"
  *and* "should a trailing space be added" *and* "should files
  also be completed as a fallback" *and* "should order be
  preserved." These are orthogonal — a completion function might
  return candidates that should suppress file completion but
  allow trailing space, or vice versa. A single-value enum
  cannot express this cross-product; a bitset can.

- **V2 bash exists because V1 cannot support newer features.**
  V1's architecture computes completions via shell-script
  fragments that pre-parse the output. V2 keeps the shell
  script simple and pushes complexity into `__complete`. This
  lets V2 support completion descriptions (shown inline with
  each candidate in terminals that support it), ActiveHelp
  messages, and ordered completions. V1 is kept for bash-3
  environments and for users who already ship V1 scripts.

- **V1 is not deleted because shell scripts get installed,
  sourced, and never re-run.** Users install completion via
  `myapp completion bash > /etc/bash_completion.d/myapp` once.
  The installed script sits on their system for years. Removing
  V1 would break every already-installed script. The cost of
  maintaining V1 (706 lines, rarely changed) is less than the
  cost of breaking downstream users.

- **`__complete` is hidden because users shouldn't call it
  directly.** It's a protocol endpoint consumed by shell
  scripts, not a user-facing command. Making it hidden
  (`command.go:232-233`, `Hidden: true`) keeps it out of
  `--help` output. Making it non-hidden would clutter user-
  visible help with an incomprehensible subcommand.

- **`DisableFlagParsing: true` on `__complete` is essential.**
  The args to `__complete` include the user's partially-typed
  flags, some of which may not be syntactically valid yet
  (mid-typing). Letting pflag try to parse them would fail
  with errors like "unknown flag: `--foo-par`." Cobra needs to
  see the raw args so it can determine which flag is being
  completed. Turning off flag parsing for this subcommand
  alone is the fix.

- **`ValidArgs` vs. `ValidArgsFunction` is the static/dynamic
  split at the user-API level.** A command with a known fixed
  set of positional values (`start`, `stop`, `restart`) uses
  `ValidArgs: []cobra.Completion{"start", "stop", "restart"}`.
  A command with dynamically-determined values uses
  `ValidArgsFunction`. Mixing would create semantic
  ambiguity, so cobra enforces only-one.

- **`ActiveHelp` is a completion channel, not a help channel.**
  It surfaces contextual guidance (`"Note: this command is
  deprecated"`, `"Hint: pass a cluster name"`) during
  completion so users see it when they're about to make a
  choice, not after they've already submitted. This is a
  UX innovation that required shell-adapter support — only V2
  bash, zsh, fish, PowerShell surface it.

- **Alternatives explicitly rejected:**
  - *Static shell scripts only.* Fails the dynamic-completion
    requirement.
  - *One generator, one script format targeting all shells.*
    No such format exists; shell completion APIs diverge too
    far.
  - *Exec an external "completion server" binary.* Extra
    binary to distribute and install; fails for
    single-binary deployments.
  - *Run completion in-process via a shared library.* No
    portable shared-library contract for bash/zsh/fish/PS.
  - *Replace V1 bash with V2 only.* Breaks existing
    installed scripts in bash-3 environments.

## Consequences

- **Every CLI built with cobra is its own completion engine.**
  Users who install the binary get completion for free by
  running `myapp completion <shell> > ...`. The binary is
  self-describing. This is a huge UX win and the main reason
  cobra's completion story is a product differentiator.

- **TAB runs the user's Go code.** If `ValidArgsFunction`
  makes a network call, TAB makes a network call. Completion
  functions should be fast and cache-friendly; a slow
  completion function makes the CLI feel broken even when
  main commands are fast. CLIs with expensive completion
  should implement local caching (filesystem cache with TTL).

- **Completion code paths are security-sensitive.** The
  completion function runs with the same privileges as the
  user's shell. A bug in completion — infinite loop, memory
  leak, accidental side effect — fires on every TAB press.
  Completion functions should be side-effect-free and bounded.

- **V1 and V2 bash diverge in feature support.** A CLI whose
  documentation says "install completion with `myapp
  completion bash`" should specify which version; users on
  bash-3 must install V1, users on bash-4+ can install V2.
  This is a documentation burden on every cobra-based CLI
  with serious bash users.

- **`__complete` subcommand is a protocol contract.** The
  output format of `__complete` is consumed by the generated
  shell scripts. Changing the output format is a breaking
  change to every installed completion script, regardless of
  the binary version. Cobra must evolve `__complete`'s output
  very carefully — introducing new directives via opt-in
  bits, never changing existing ones.

- **Users can accidentally install the wrong completion script
  version.** A CLI that was built with cobra v1.5 and emits
  V2 scripts, run against a shell that sourced a V1 script
  from cobra v1.0, produces confusing partial-completion
  behavior. Cross-version script compatibility is a support
  burden.

- **`DisableFlagParsing` on `__complete` means flag-group
  validation doesn't run there.** Completion-time "which flags
  are valid given the current input" is computed by cobra's
  own logic, not by the pipeline. The rule for what
  `enforceFlagGroupsForCompletion` does (`flag_groups.go:225-
  290`) is coupled to the annotation format, which must be
  kept in sync.

- **Binary size includes the completion engine.** Every cobra
  CLI bundles the completion logic whether or not the binary
  will ever be used interactively. For very size-sensitive
  deployments this is a cost. `CompletionOptions` can disable
  the completion subcommand but the completion code is
  still linked.

- **`__complete` must handle every legal subcommand path.**
  Every subcommand, every flag, every alias must be reachable
  through the completion protocol. Adding a command without
  registering its completion behavior leaves completion silent
  for that command. Code review of new commands should check
  that `ValidArgs` or `ValidArgsFunction` is appropriate.

- **Shell-specific bugs are shell-specific.** A completion
  bug that manifests only in PowerShell's `Register-Argument-
  Completer` interaction with the generated script is hard to
  diagnose from a platform-neutral vantage point. Cobra's
  test suite includes per-shell tests (e.g.,
  `bash_completionsV2_test.go`, `zsh_completions_test.go`,
  etc.) but real-shell coverage is limited.

- **`ShellCompDirectiveError` exists to surface programming
  errors.** A completion function that computes nothing and
  returns `ShellCompDirectiveError` causes the shell to do
  nothing (not fallback to file completion). This is the right
  choice for "something went wrong in the completion function"
  but subtly different from "no candidates, please fall back
  to file completion" (`ShellCompDirectiveDefault`).

- **Completion of mutually exclusive flags filters via
  `enforceFlagGroupsForCompletion` (ADR-06).** The flag-group
  annotations do double duty: they validate post-parse, and
  they prune completion candidates pre-parse. Contributors
  changing either code path must remember the other exists.

- **The generated shell scripts carry protocol knowledge.**
  The bash-V2 script, for example, knows to split the
  completion output on a specific delimiter and interpret
  the `:` separator for descriptions. Changing the
  `__complete` output format requires regenerating and
  re-distributing scripts to users. Releases that change the
  protocol are effectively major-version bumps for the
  completion feature even when they're minor bumps for the
  library.

- **New shells are add-a-generator.** Adding support for, say,
  nushell means writing a generator that emits a nushell-
  compatible script which calls `__complete`. The completion
  engine itself doesn't need to change. This is the cleanest
  possible contract for shell support: the engine is fixed,
  the adapters are per-shell.
