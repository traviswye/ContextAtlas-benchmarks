## §4 — H1 prompt verbatim (post-refinement)

Snapshot of `EXTRACTION_PROMPT` as committed in main repo
`bb5efbe`. Authoritative source: `src/extraction/prompt.ts`.

```
You are extracting architectural claims from the input below.

Given the input, extract architectural constraints, preferences, and contextual information present in the prose. Output strictly valid JSON matching this exact schema:

{
  "claims": [
    {
      "symbol_candidates": ["string array of class/function/module names referenced in the prose"],
      "claim": "concise statement of the constraint or fact",
      "severity": "hard" | "soft" | "context",
      "rationale": "why this matters, from the input",
      "excerpt": "short verbatim quote from the input supporting this claim"
    }
  ]
}

Severity taxonomy:
- "hard": explicit constraint, violation is a bug. Signaled by:
  - Mechanical markers: "@deprecated" tag (JSDoc), "Deprecated:" line prefix (godoc), ".. deprecated::" directive (Sphinx)
  - Prose patterns: "must", "MUST", "never", "always", "required", "not allowed" — but only when the prose asserts a constraint on the consumer (e.g., "Implementations MUST handle nil context"). API documentation describing how a library works (e.g., "Cobra requires you to define X") is descriptive, not assertive — default to context.
- "soft": preference or recommendation. Signaled by "should", "prefer", "avoid", "generally", "recommended", "Notice that...", or descriptive cautions ("can be dangerous", "may cause"). Imperative procedural guidance ("Set this to X") is also soft.
- "context": background information or rationale; no rule asserted. Descriptions of why things exist, how they work, or what something is. DEFAULT category for descriptive prose without imperatives.

When mechanical severity signals are absent — for example, Python docstrings often communicate deprecation only via runtime warnings.warn() calls, not in the docstring text itself — do not over-extract hard severity from descriptive prose. Default to context unless prose contains explicit recommendation language.

Skip non-architectural content: YAML frontmatter, license headers, installation instructions, changelogs, deployment steps, version markers, and pure type-shape annotations (e.g., "@param T - The type of X", "@returns Response" without architectural rationale, "method must be one of GET, OPTIONS, HEAD, POST, PUT, PATCH, or DELETE" — enum-of-valid-values is type-shape, not architectural). However: "@param verify - Either True to use SSL context with default CA bundle, False to disable verification" IS architectural — the parameter encodes a security default, not just a type.

If the input contains no architectural claims — for example, a terse implementation contract like "Check if the client is closed" or pure behavioral description without architectural rationale — output {"claims": []}. Do not invent claims to fill output.

For symbol_candidates: extract class/function/module/package names referenced in the prose. For inputs that are docstrings attached to a specific symbol, the documented symbol itself need NOT appear in symbol_candidates (provenance carries that); include only OTHER symbols mentioned.

For external documentation references (Markdown reference links like [Authentication][0] with URL definitions, JSDoc {@link URL} tags, "See also: ..." prose): preserve the human-readable label in the claim text but do not include URLs that would be meaningless out of their original context.

Output ONLY the JSON object. No prose, no markdown fencing, no commentary.

Input:
---
```

**Refinement-driven additions vs original ADR-only baseline:**
- Mechanical severity markers (`@deprecated` / `Deprecated:` /
  `.. deprecated::`) — explicitly listed in hard taxonomy
- API-documentation vs assertion distinction in prose-hard
  guidance (Refinement 2)
- Soft signals from docstring prose ("Notice that...",
  "can be dangerous", "Set this to X" procedural)
- Zero-mechanical-signal paragraph (Python deprecation
  runtime-only)
- Skip-content expanded (license headers, version markers,
  type-shape annotations including enum-of-valid-values per
  Refinement 1, with verify-parameter positive contrast)
- Zero-claim discipline paragraph
- `symbol_candidates` semantics for docstrings (provenance
  carries documented symbol)
- External documentation references (preserve human-readable
  label, drop context-meaningless URLs)
- Header generalized "from a document" → "from the input"

## §8 — Refinement findings (verbatim before/after)

### Refinement 1 — Sample #5 enum over-extraction

**Before refinement.** Pre-refinement Sample #5 produced 3
claims; one was the enum-of-valid-values:

```json
{
  "symbol_candidates": [],
  "claim": "method must be one of GET, OPTIONS, HEAD, POST, PUT, PATCH, or DELETE",
  "severity": "context",
  ...
}
```

This is type-shape (the parameter's enumerated valid values),
not architectural. The original prompt's skip-content guidance
listed pure type-shape examples ("@param T - The type of X",
"@returns Response") but didn't generalize to enum cases —
extraction extracted the enum as if it were a meaningful
constraint claim.

**Refinement 1 applied to prompt.** Added enum-of-valid-values
explicit example to the skip-content paragraph:

> ...pure type-shape annotations (e.g., "@param T - The type of
> X", "@returns Response" without architectural rationale,
> **"method must be one of GET, OPTIONS, HEAD, POST, PUT, PATCH,
> or DELETE" — enum-of-valid-values is type-shape, not
> architectural**).

**After refinement.** Re-run produced 2 claims, both context,
with the enum claim eliminated:

```json
[
  {
    "symbol_candidates": ["ssl.SSLContext"],
    "claim": "verify parameter accepts True (default CA bundle), False (disable verification), or an ssl.SSLContext instance for custom context",
    "severity": "context",
    ...
  },
  {
    "symbol_candidates": [],
    "claim": "trust_env controls whether environment variables are used for configuration",
    "severity": "context",
    ...
  }
]
```

Refinement 1 stuck. The verify-parameter claim demonstrates
Refinement B's positive-contrast example ("@param verify... IS
architectural") was simultaneously internalized — the model
correctly distinguished enum-skip from
architectural-with-type-context retention.

### Refinement 2 — Sample #10 prose-hard misfire

**Before refinement.** Pre-refinement Sample #10 produced 2

{

  "claim": "Cobra requires usage and description to be defined as part of a command definition",

**Refinement 2 applied to prompt.** Added API-documentation vs

> Prose patterns: "must", "MUST", "never", "always", "required",

> on the consumer (e.g., "Implementations MUST handle nil

```json
{
  "symbol_candidates": ["Cobra"],

```

Refinement 2 stuck. The phrase "Descriptive API contract of

## §9 — Coverage gap (Step 10/11 informer)

**No sample tests prose-hard severity unambiguously on this
sample set.**

After Sample #10 was reframed as over-detection-discipline test
(per §3 pre-registration correction), every severity=hard
expectation in the calibration set is mechanical:
- Sample #2: `@deprecated` JSDoc tag (mechanical)
- Sample #9: `Deprecated:` godoc line prefix (mechanical)

True prose-hard — docstrings asserting consumer-facing
constraints like "Implementations MUST handle nil context" or
"Callers must call Close() before destruction" — is **absent
from cobra/hono/httpx Step 8 probe samples**. These libraries'
prose tends toward descriptive ("Cobra requires...", "must be
one of...") rather than assertive ("you MUST handle nil
context").

**Implication for H1 vs H2 calibration:** Calibration could not
empirically test prose-hard detection accuracy. H1 prompt's
prose-pattern guidance ("must" / "MUST" / "never" / etc.) is
specified but unverified on this sample set. Whether H1
correctly detects prose-hard on real consumer-asserting
docstrings is **empirically open**.

**Implication for Step 10/11 sample selection:** target
docstrings with explicit `MUST` / `always` / `required`
constraint assertions (likely from libraries with
contract-heavy APIs — e.g., interfaces with
implementation-MUST clauses, security-critical APIs with
caller-MUST guarantees). Validate prose-hard detection
empirically before relying on it at scale.

**Implication for Step 14/15 reference run criteria:**
prose-hard detection accuracy could be a separate measurement
axis; if reference runs surface prose-hard misclassification,
prompt refinement is the lever (Refinement 2 pattern) rather
than architectural switch.

## §10 — Open questions deferred to Steps 10/11

Five open questions surfaced during Step 9 calibration; all
deferred to Step 10/11 implementation as inheritance:

**(a) AST-level Python deprecation as separate claim source.**
Probe §6 / §10 inheritance preserved. Python deprecation is
runtime-only via `warnings.warn(DeprecationWarning)`; static
docstring extraction cannot detect. Step 10/11 should decide
whether `warnings.warn(...)` AST detection is in scope as a
separate claim source (with `severity=hard`), or out-of-scope
for v0.3 Stream B (defer to v0.4 backlog as new Theme item).

**(b) Module-level Python SymbolId shape.** Probe §10 deferred:
`sym:py:<path>:<module>` vs `(module)` reserved name. Calibration
sample set didn't include module-level docstrings (httpx
`__init__.py` is import-only); evidence remains thin. Step 10/11
implementation must lock the convention; recommend evaluating
on broader Python repo (not just httpx) before locking.

**(c) TypeScript extraction path** (tsserver `quickInfo` /
`textDocument/hover` vs direct AST via
`ts.getJSDocCommentsAndTags()`). Probe §8.2 leaned tsserver
hover for adapter-as-LSP-only design. Step 11 should verify
whether tsserver hover preserves enough JSDoc tag granularity
for severity inference (`@deprecated` detection); if
granularity insufficient, fall back to direct AST parse.

**(d) Pipeline integration architecture** (per-symbol vs
batched API calls). Calibration used per-docstring single-call
shape; production Steps 10/11 may want per-file or per-class
batching for cost amortization. Architecture decision is
pipeline-design territory; calibration findings don't constrain
