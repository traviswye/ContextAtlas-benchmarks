### 1.1 Multi-symbol `get_symbol_context` call shape

**Source:** Phase 7 §5.1 (Go grep-ability paradigm sensitivity).
Already promoted to ROADMAP.md v0.3 deliverables.

**Scope:** `get_symbol_context` currently takes one symbol at a
time. On flat-package languages (Go), knowledgeable Grep
retrieves multiple related symbols in one regex disjunction; CA's
per-symbol fetches add structural overhead. A multi-symbol call
shape (or batched `find_by_intent` with explicit symbol
disjunction) closes most of the gap.

**Relevance to v0.3:** **High.** Phase 7 §5.1 mechanism is
load-bearing — without this, CA continues to lose on c4-class
prompts when Go (and presumably other flat-package languages)
benchmarks expand in step 13.

**Touch points:** MCP tool interface (changes the shape Claude
Code sees), ADR-N for the new tool variant or option.

### 1.2 Claim attribution + ranking precision

**Source:** [`atlas-claim-attribution-ranking.md`](atlas-claim-attribution-ranking.md).
Status: Candidate. Filed Phase 6 §5.1 (httpx p4-stream-lifecycle
investigation, 2026-04-24).

**Scope:** Three gaps surfaced from p4 investigation:
- **Claim-attribution inheritance:** every claim from an ADR
  inherits its frontmatter-resolved symbols as baseline; per-
  claim candidates rarely add new symbols. Result: many claims
  attached to the same baseline symbol set; per-symbol lookup
  returns muddy bundles.
- **Per-symbol claim ranking is deterministic, not query-aware:**
  `get_symbol_context` returns claims in insertion/ID order. For
  symbols with 17+ claims attached, the "first claim shown" is
  arbitrary relative to query intent.
- **Tool asymmetry:** `find_by_intent` uses BM25 ranking
  (ADR-09); `get_symbol_context` and `impact_of_change` don't.

**Three candidate fixes** (note proposes priority order):
1. ADR authoring validation at extraction time (small, safe,
   ~30–50 LOC).
2. Narrower claim attribution — drop frontmatter-baseline
   inheritance (~20–30 LOC; needs benchmark evidence).
3. Symbol-scoped claim ranking (BM25 or embedding-based;
   medium-large; tool-interface change).

**Relevance to v0.3:** **High.** Touches the same surface as
1.1 (the bundle's content shape). Coupling: Stream C
(docstring/README mining) alone won't fix this — adding more
claim sources without ranking precision would worsen the
problem.

**Touch points:** Extraction pipeline (Fix 1), MCP tool surface
(Fix 3 — likely). Possible ADR amendment for ADR-09 or new ADR
for symbol-scoped ranking.

### 1.4 Production-correctness findings — dogfood test cycle (2026-04-26)

**Source:** v0.3 dogfood test cycle (prep gate-check, Round 1+2
explanation tasks, Round 3 A/B code-change trial). Ten
production-correctness items surfaced across extraction
pipeline, documentation, MCP config, FTS5 tokenizer, and
Windows developer experience. Items 1.4.1-1.4.3 are
extraction-pipeline gate-check findings (initial Theme 1.4
filing, commit `1f83112`); 1.4.4-1.4.10 extend the inventory
with later dogfood-cycle findings.

- **1.4.1 — Stale-SHA-on-skip.** When extraction hits
  `max_tokens` on a doc, atlas retains prior
  `source_shas[doc]` + prior claims. Atlas falsely claims
  doc is current. Should record actual SHA +
  `skipped_in_last_extraction` flag. **Priority: v0.3.**
  Hit on DESIGN.md during prep.
- **1.4.2 — `EXTRACTION_MAX_TOKENS` configurability.**
  Hardcoded at `src/extraction/prompt.ts:95` (bumped
  8000 → 16000 during prep to fit DESIGN.md post-Phase-3-D2).
  Should be `.contextatlas.yml` config or CLI flag.
  **Priority: v0.4** unless Step 14 surfaces need at scale.
- **1.4.3 — max_tokens-skip silent-failure-exit.** Skip
  warns at `[warn]` level; exit zero; metrics show
  `extraction_errors=0`. Caller cannot detect the failure
  mode without gate-check. Should emit `max_tokens_skips=N`
  in final metrics + non-zero exit option. **Priority: v0.3.**
  Compounds with 1.4.1.
- **1.4.4 — `symbols_indexed` metric inconsistency.**
  Extraction CLI reports `symbols_indexed=N`; committed
  atlas's symbols array has N-3. Likely pre-pruning count or
  includes unresolved candidates not in final atlas. Caller
  cannot trust metric reflects committed state. Surfaced
  during atlas commit gate-check (commit `1a116a0`).
  **Priority: v0.4.**
- **1.4.5 — README §172 MCP config example incorrect.**
  `args: ["serve"]` failed because the binary doesn't have a
  `serve` subcommand; default no-arg invocation works.
  Surfaced during dogfood test setup. **Status: CLOSED via
  main repo `d25835f`** (two-example structure: global PATH
  + abs-path fallback).
- **1.4.6 — FTS5 tokenizer drops identifier-shaped feature
  names.** unicode61 splits `narrow_attribution` → `narrow` +
  `attribution`; LLM agents querying by feature name fail
  silently. Surfaced during Round 2 dogfood Q2 (Round 1+2
  evidence note `177bdb5`). **Status: CLOSED via main repo
  `cb6df70`** — Round 3 A/B trial arm-B precision-first
  design adopted; ADR-17 added
  (`fts5-identifier-aware-tokenizer`); schema migration v5
  (drop+recreate `claims_fts` with `unicode61 tokenchars '_-'`
  + index-time dual-form indexing).
- **1.4.7 — BM25 server flag default-off creates
  query-parameter silent ignore.** Caller passes `query`
  parameter to `get_symbol_context`; flag default-off
  (Reading 3 ship state); query accepted but ignored. Caller
  has no signal BM25 didn't activate. Surfaced during Round 2
  dogfood Q3 (Claude Code detected v0.2 chain ordering in
  returned claims and inferred flag was off). Should either
  error loudly OR document in tool description.
  **Priority: v0.4** (depends on Step 7 ship default
  reconsideration; if Fix 3 default-on lands at v0.4, this
  finding becomes moot).
- **1.4.8 — Missing `.gitattributes` causes Windows clone
  test failures.** Travis observed test failures on Windows
  clones during Round 3 A/B trial setup attributable to
  line-ending normalization conflicts. Without explicit
  text/binary classifications in `.gitattributes`, Windows +
  Linux developer environments round-trip text files
  differently. The `warning: LF will be replaced by CRLF`
  warnings throughout the v0.3 dogfood session indicate the
  gap. Should land `.gitattributes` with text/binary rules
  matching repo conventions. **Priority: v0.3** (blocks
  external Windows contributors).
- **1.4.9 — Identifier query recall trade-off (1.4.6 design
  consequence).** Arm B's precision-first design (commit
  `cb6df70`) excludes split-word-only docs from
  identifier-form queries. Doc that writes "narrow
  attribution" as natural words won't match query
  `narrow_attribution`. Trade-off intentional; documented in
  ADR-17 + 1.4.6 commit message. Hybrid approach available
  (Arm B's index-time dual-form + Arm A's query-time
  joined-identifier expansion only for multi-token queries)
  if recall failures surface. **Priority: v0.4. Revisit
  trigger:** Stream D measurement OR production usage
  surfacing recall failures on identifier-form queries
  against split-word docs.
- **1.4.10 — TypeScript adapter diagnostics tests fail on
  Windows.** `src/adapters/typescript.test.ts:124`
  (TypeScriptAdapter getDiagnostics expects errors from
  broken fixture, gets 0) and `src/adapters/conformance.test.ts`
  (line 201 region, similar pattern) fail consistently on
  Windows due to `typescript-language-server` LSP behavior
  differences. Tests pass on Linux/macOS dev environments.
  Surfaced during v0.3 dogfood Round 3 synthesis verification
  (`cb6df70` land verification). Affects Windows contributors
  during local test runs. **Priority: v0.4** (developer
  experience; not blocking external release).

## Sequencing constraint — Theme 1.2 prerequisites Stream C

Theme 1.2 (claim attribution + ranking precision) and the
original v0.3 Stream C work (docstring extraction, README /
PR / commit-message mining) are sequence-coupled. Adding claim
sources without first sharpening attribution + ranking would
**amplify** the muddy-bundle problem Phase 6 §5.1 identified
on p4-stream-lifecycle.

**Mechanism.** Stream C multiplies claims-per-symbol because
new sources naturally cluster around the symbols they
describe — docstrings live next to their symbol; PR
descriptions reference symbols the PR touched; commit
messages reference everything in the changeset. Per-symbol
ranking precision becomes *more* important as the claim count
grows, not less.

**Concrete v0.3 implication.** If v0.3-SCOPE includes Stream C
deliverables, it must ALSO include Theme 1.2 OR explicitly
acknowledge that the muddy-bundle problem will worsen until
1.2 lands. The original ROADMAP-named v0.3 deliverables (claim
source enrichment) and the v0.2-derived Theme 1.2 are not
independently selectable — they are sequence-coupled.

**Possible v0.3 scope shapes implied:**
- **Stream C + Theme 1.2 Fix 1 (ADR authoring validation,
  smallest):** ~30–50 LOC; doesn't fix per-symbol ranking but
  catches the worst authoring-quality drivers of the problem.
- **Stream C + Theme 1.2 Fix 2 (narrower attribution) or
  Fix 3 (symbol-scoped ranking):** larger v0.3; fixes the
  ranking precision directly.
- **Stream C only, accepting the tradeoff:** v0.3 scope doc
  must explicitly acknowledge the ranking-precision cost as a
  known regression on dispersed-claim repos.

Other coupling not flagged this strongly: 1.1 (multi-symbol
`get_symbol_context`) is independent of Stream C; 1.3 (commit
SHA) is independent. Theme 2.1 + 2.2 sequencing flagged inside
Theme 2 below.

## Theme 2 — Benchmark methodology hardening

Items that change the harness or measurement methodology rather

### 2.1 Atlas-file-visibility methodology fix

**Source:** Phase 7 §5.2 (cobra c6 measurement artifact).
**Note still to be authored** at
`research/atlas-file-visibility-benchmark-methodology.md`. Already
promoted to ROADMAP.md v0.3 deliverables.

**Scope:** Visible `atlases/<repo>/` artifacts in the benchmarks
workspace can mislead the beta condition on prompts whose target
symbol has a generic name (`Execute` in cobra, hypothetically
`get`/`set`-class symbols in other targets). Beta tries to
interrogate the atlas data files instead of source. Three
candidate fix paths:
- **Trace-time filter** (lowest-cost; recommended starting
  point): exclude cells where beta's trace references atlas
  paths.
- **Clean-workspace mode** (longer-term): run beta against a
  copy of `repos/<repo>/` only, with no parent-directory
  visibility into atlases.
- **Atlas-aware prompts** (discouraged): would require
  re-baselining all prior phases.

**Relevance to v0.3:** **High.** Currently the only known
beta-condition contamination pattern. Step 13 expansion will
amplify exposure unless v0.3 lands the fix.

**Touch points:** Harness code (`src/harness/run.ts` or
similar). Possibly an RUBRIC.md amendment for the methodology
addition.

**v0.3 prep action:** the source note itself needs authoring
during v0.3 execution rather than at retro time, per the v0.2
retrospective lesson.

### 2.3 Cost priors — Go-specific bucket scale

**Source:** Phase 7 §8.4. Extracted from synthesis to inventory
during the v0.3 backlog walk; not previously filed as a
standalone backlog note.

**Scope:** Cost priors in `src/harness/run.ts` currently use
uniform per-bucket multipliers regardless of target language.
Phase 7 cobra data shows Go's grep-ceiling effect lowers
alpha-condition costs ~40% vs hono on equivalent buckets
(cobra alpha total $2.73 vs hono alpha total $7.25, 6 prompts
each). Refactor to support per-language bucket scales; Phase 7
data is sufficient to seed Go priors. Revised cobra-derived
projections ($0.30/cell vs blended hono+httpx $0.61/cell) feed
Phase 7 §7's revised step-13 budget envelope of $115–150
(superseding earlier $176–210).

**Relevance to v0.3:** **Medium.** Not load-bearing for any
v0.3 deliverable. Without recalibration, v0.3 reference run
budget projections inherit hono-biased priors. Cleanest if
bundled with whatever harness work v0.3 needs anyway (likely
Theme 2.1's atlas-file-visibility fix).

**Touch points:** `src/harness/run.ts` `COST_PRIORS_V0_1`
constant. Possibly RUBRIC.md update documenting the
per-language calibration pattern.
