## TL;DR — decision contract

Round 3 tested code-change task value proposition (the Round 1+2
explanation-task evidence's untested complement) by giving each
arm the same task: locate the FTS5 identifier-shape bug
(`narrow_attribution` miss documented as Theme 1.4.6 in Round 2),
design and implement a fix, write tests, and document via ADR.
Identical 3-paragraph prompt; only setup variable was MCP
availability.

**Selection: Arm B's index-time dual-form approach.** Three-bullet
rationale:

1. **Thesis alignment.** ContextAtlas's pre-extracted-claims-with-
   structural-attribution premise IS a precision-optimization
   pattern. Arm B's identifier-form-only matching aligns; Arm A's
   recall-first OR-expansion fights it.
2. **Empirical evidence.** Round 2 Q2's failure case (Claude Code
   querying `narrow_attribution`, missing the canonical claim) is
   what Arm B fixes precisely. Arm A fixes the same case but with
   noise (BM25 inflation from split-part hits competing with
   canonical-identifier docs).
3. **Recall trade-off captured.** Arm B may miss spaced-form-only
   docs queried with identifier form. No production evidence yet.
   Hybrid approach (Arm B's index-time dual-form + Arm A's
   query-time joined-identifier expansion for multi-token cases)
   captured as Theme 1.4.9 revisit trigger if production usage
   surfaces recall failures.

**Variance disclaimer:** 19% wall-clock + 12% token premium
concentrated in step-4 self-correction is single-trial variance
at N=1, not generalizable cost. See Section 11 for full caveats.

## Task prompt design

**Final task prompt (verbatim, identical across both arms):**

> There's a bug in the FTS5 tokenizer configuration: identifier-
> shaped feature names like narrow_attribution become narrow +
> attribution when indexed in claims_fts. This means
> find_by_intent queries for those names return silent search
> misses even when the content is indexed. Fix the tokenizer
> configuration so that identifier-shaped names are searchable
> as units (start with narrow_attribution-style underscored
> names; investigate whether camelCase, kebab-case, or other
> shapes have similar issues and address them if they do).
>
> Preserve existing FTS5 behavior on natural-language queries —
> search relevance on phrases, multi-word queries, and partial
> matches should not regress. Add tests covering both the new
> tokenization behavior (regression test for the failure mode)
> and the preserved natural-language behavior (canary test).
>
> Document the new tokenizer config + rationale in an ADR
> (amend ADR-09 or new ADR — your call based on scope of
> change).

**Discipline-justified refinements vs Travis's draft prompt:**

(a) **Scope softening on identifier shapes.** Original draft
specified snake_case + kebab-case explicitly; final softened to
"identifier-shaped" letting agent decide unicode61-tokenchars
scope. Both arms landed on `'_-'`.
(b) **Preservation specificity.** Final prompt explicitly names
phrases, multi-word queries, and partial matches to prevent
recall regression on natural-language queries.
(c) **Test-shape canary discipline.** Final prompt explicitly
names regression test (failure mode) AND canary test (preserved
behavior); both arms added 3-4 new canary tests for identifier-
shape coverage.
(d) **ADR-09 amendment vs new ADR — agent's call.** Both arms
chose new ADR-17 with forward pointer to ADR-09; neither chose
amendment.

**Methodological notes:** prompt does NOT mention MCP, atlas,
or ContextAtlas tooling. **Asymmetry-protected** — Arm A has no
signal that this is an MCP-availability A/B trial. Both arms
see only "FTS5 identifier-shape tokenizer bug, fix it."

v2.1.118 Claude Max). No warm-up prompt. No `.mcp.json` in

step 4: 40.6k; step 5: 49.5k.

round.

**Implementation concentration.** `find-by-intent.ts` +65 LOC
(query-time expansion in `buildMatchQuery`). Storage layer
unchanged beyond schema migration v5 (drop+recreate
`claims_fts` + triggers).

+406 / -18.

**ADR-17 shape.** 223 LOC. Five paragraphs alternatives-considered.
Worked examples but no actual-atlas verification. Camel case

## Comparative structural analysis

| Dimension | Arm A | Arm B |
|---|---|---|
| Files modified | 6 | 6 (same set) |
| New ADR file | `ADR-17-fts5-identifier-tokenizer.md` | `ADR-17-fts5-identifier-aware-tokenizer.md` |
| Total LOC delta (excl. package-lock) | +406 / -18 | +428 / -11 |
| ADR-17 LOC | 223 | 294 |
| Implementation LOC concentration | `find-by-intent.ts` (+65) | `db.ts` (+90) |
| Test LOC | `find-by-intent.test.ts` (+219), `db.test.ts` (+69) | `find-by-intent.test.ts` (+248), `db.test.ts` (+40) |

**Architectural divergence (load-bearing).** Both arms made the
same schema/sanitizer/tokenizer/migration choices: tokenchars
`'_-'`, sanitizer preserves `_` and `-` in queries, schema
migration v5 (drop+recreate `claims_fts` + triggers), camelCase
deferred. The divergence is in cross-form coverage mechanism:

| Mechanism | Arm A (query-time expansion) | Arm B (index-time dual-form) |
|---|---|---|
| `buildMatchQuery(["narrow_attribution"])` | `"narrow_attribution" OR "narrow" OR "attribution"` | `"narrow_attribution"` (phrase only) |
| Trigger inserts | raw claim/rationale/excerpt | dual-form: original + `_`/`-`-replaced-with-space concat |
| Index for doc `"narrow_attribution flag"` | tokens: `[narrow_attribution, flag]` | tokens: `[narrow_attribution, flag, narrow, attribution, flag]` |
| FTS5 `-` NOT-operator handling | universal quoting | selective (hyphen-only) quoting |

**Implementation-LOC inversely correlates with architectural
choice.** Arm A puts logic in the query layer (+65 in
`find-by-intent.ts`); Arm B puts logic in the storage layer
(+90 in `db.ts`). Different complexity location; same end
behavior on most queries — confirms the architectural divergence
is real, not cosmetic.

## Cross-arm test results

Both arms' test suites ran against the other's implementation.
Most "failures" are architectural-difference (tests assert each
arm's specific shape choices), not behavioral bugs. Two are real
behavioral differences.

**Arm B's tests on Arm A: 8 fail / 35 pass.**

| Failure | Type | Notes |
|---|---|---|
| `buildMatchQuery(["find-by-intent"]) === '"find-by-intent"'` | architectural diff | A returns expanded form |
| `buildMatchQuery(["per-symbol", "claims"])` | architectural diff | A's expanded form differs |
| `buildMatchQuery(["narrow_attribution"]) === '"narrow_attribution"'` | architectural diff | A returns expanded form |
| `fts5vocab` dual-form check | architectural diff | A doesn't index dual-form |
| `db.test.ts` "tokenize=" string | test brittleness | whitespace differs |
| `db.test.ts` dual-form vocab | architectural diff | A doesn't dual-form |
| **kebab-case query hits canonical claim** | **real behavioral** | A returns "Other" instead of "KebabFeature" |
| Same kebab-case test variant | follows from above | |

**Arm A's tests on Arm B: 5 fail / 36 pass.**

| Failure | Type | Notes |
|---|---|---|
| joined-identifier expansion `["payment", "idempotency"]` | architectural diff | B doesn't expand at query |
| split-parts expansion `["narrow_attribution"]` | architectural diff | B returns phrase only |
| split-parts expansion `["drop-with-fallback"]` | architectural diff | B returns phrase only |
| dedup test `["A_a"]` | architectural diff | B doesn't dedup |
| **identifier-shaped query finds split-words doc** | **real behavioral** | B returns 0 matches; A's test expects 1 |

**Two real behavioral differences:**

1. **Arm A over-recalls on identifier queries with overlapping
   noise docs.** Test: query `find-by-intent` against (a) doc
   "find-by-intent tool is a thin composite" and (b) doc "Find
   the right symbol by intent rather than name." Arm A's expansion
   adds `find`, `by`, `intent` to OR clauses. Doc (b) hits all
   three split parts; doc (a) hits only the canonical phrase.
   BM25 ranks doc (b) higher. Canonical-identifier doc loses to
   component-word noise.
2. **Arm B excludes split-word-only docs from identifier queries.**
   Test: query `narrow_attribution` against doc "narrow
   attribution mode is the default" (no underscore). Arm A's OR
   fallback `"narrow" OR "attribution"` matches; Arm B's
   phrase-only `"narrow_attribution"` doesn't. Arm B intentionally
   treats identifier-form and split-word as semantically different.

**Neither difference is a bug.** Different empirical
interpretations of "what should an identifier query match?"

## Decision rationale (Travis's four axes)

Selection of Arm B for 1.4.6 fix landing was made on four axes
explicitly weighed at decision time:

**(1) Thesis alignment.** ContextAtlas's pre-extracted-claims-with-
structural-attribution premise IS a precision-optimization
pattern. The whole point of paying extraction-time API cost is
to enable query-time precision — narrow, structurally-attributed
claims rather than broad keyword recall. Arm B's identifier-form-
only matching aligns with this premise. Arm A's recall-first
OR-expansion fights it (BM25 inflation from split-part hits is
exactly the noise the extraction premise is meant to suppress).

**(2) Empirical evidence.** Round 2 Q2's failure case (Claude
Code querying `narrow_attribution`, missing the canonical claim
documented in Round 1+2 evidence note) is what Arm B fixes
precisely. Arm A fixes the same case, but the canonical-
identifier doc must compete with split-part-hit noise docs in
BM25 ranking — solving the problem with side-effects rather than
solving it cleanly.

**(3) Recall trade-off hypothetical.** Arm B's known limitation:
identifier-shaped queries don't match split-word-only docs. No
production evidence this happens. Captured as **Theme 1.4.9
revisit trigger** in benchmarks `a98c63d`: if production usage
surfaces recall failures from this design choice, hybrid approach
available (Arm B's dual-form indexing + Arm A's joined-identifier
expansion only for multi-token queries where dual-form doesn't
help).

**(4) Premium reframing.** Arm B's 19% wall-clock + 12% token
premium concentrated in step-4 self-correction. Whether this
represents MCP-augmented development cost (warmed session more
rigorous → more thorough test discovery → caught case Arm A
might have missed implicitly) OR trial-specific variance
(task-shape sensitivity, agent response variability) is
empirically indeterminate at N=1. **Honest framing: trial-
specific variance, not generalizable cost.** Whether the premium
is real and worth paying is a Stream D measurement question, not
a Round 3 conclusion.

## Implications for v0.4 Fix 3 reconsideration

Round 3 evidence informs but doesn't lock v0.4 planning. Three
implications:

**(a) Arm B's precision-first thesis alignment validates BM25
design's precision intent.** ADR-16 BM25 design prioritizes
precision over recall (severity-first chain α with selective
cross-severity promotion). Arm B's selection on thesis-alignment
grounds reinforces that precision-optimization is the
ContextAtlas value proposition. Supports v0.4 Fix 3 default-on
**if discovery engineering is bundled** (Round 1+2 evidence's
binding constraint per Layer 1 framing).

**(b) Recall trade-off (1.4.9) becomes Stream D measurable axis.**
The hypothetical recall failure case (identifier query →
split-word doc) is a concrete measurement target for Stream D
Step 14/15 cross-language reference runs. If beta-ca measurement
across hono/httpx/cobra surfaces queries where this trade-off
matters, hybrid approach moves from contingent revisit to v0.4
priority.

**(c) Round 4 protocol proposal.** BM25 server flag explicitly
enabled (`mcp.symbol_context_bm25=true`) + same explanation-
question prompts (Round 1+2 set) to test BM25 reordering
effectiveness end-to-end. **NOTE: Round 4 closes the Layer 3
(server gating) evidence gap from Round 1+2, not the code-
change task value-prop gap (which Round 3 closed for
explanation-vs-code-change distinction).** Both evidence gaps
remain open at v0.3 ship; Round 4 + Stream D beta-ca measurement
together would resolve them. **Capture as proposal; do not
execute today.**

## Honest framing — what Round 3 does and doesn't prove

Five caveats:

**(a) N=1 single-trial.** One developer (Travis), one repo
(ContextAtlas itself), one task shape (FTS5 tokenizer fix),
one Claude Code instance type per arm. Strong directional signal;
not statistical. Wall-clock + token premium numbers are
single-observation, not distributions.

**(b) One-task scope.** Theme 1.4.6 is structurally a small,
well-scoped storage-layer fix with clear bug reproduction. Round
3 evidence does NOT generalize to large refactors, cross-cutting
architectural changes, or tasks where MCP context might be more
or less useful. The dogfood test premise stays narrow.

**(c) ContextAtlas-on-itself substrate.** Atlas substrate is
ContextAtlas's own ADRs + DESIGN.md + STEP-PLAN docs. Arm B's
agent had access to architectural context about the codebase it
was modifying — high alignment between atlas content and task
content. External repos with less ADR-cultivated atlases may see
different dynamics.

**(d) Wall-clock + token premium = trial-specific variance, not
generalizable cost.** Per Section 8 axis (4): whether 19%
wall-clock + 12% token premium represents MCP-augmented
development cost OR task-shape variance OR Claude Code response
variability is empirically indeterminate at N=1. Stream D
measurement at scale would resolve.

**(e) Stream D measurement still load-bearing.** Round 3 informs
v0.4 planning + Theme 1.4 backlog + 1.4.6 fix selection. Stream
D Step 14/15 measure code-change-task activation across
benchmark targets. Round 3 substitutes for neither Stream D
measurement nor production-target evidence.
