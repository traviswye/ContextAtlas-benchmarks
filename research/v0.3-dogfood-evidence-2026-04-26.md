## Three-layer activation framework

Reading 3's "activation gap" decomposes into three independent layers.
Round 2 evidence resolves them differently:

**Layer 1 â€” Discovery** (does Claude Code route to MCP at all, vs
Glob/Grep/Read?). Round 1 vs Round 2 isolates this. Cold session = MCP
invisible to default tool selection; warmed session = MCP routes
naturally with appropriate tool selection. **Resolved: discovery
problem.**

**Layer 2 â€” Query-passing** (does the caller send `query` parameter
to `get_symbol_context` when relevant?). Round 2 Q1 (`query="ADR frontmatter symbols vs per-claim symbol attribution"`)
and Q3 (`query="claim ordering severity bm25 ranking truncation cap"`)
both confirmed. Multi-symbol form (Theme 1.1) also exercised
naturally on related-symbol-pair questions. **Resolved: query-passing
works.**

**Layer 3 â€” Server flag gating** (does BM25 actually fire, or is
`mcp.symbol_context_bm25` default-off per Reading 3?). Round 2 Q3
empirically caught this â€” caller passed query correctly, atlas
returned v0.2-chain-ordered claims (severity-first, no BM25
reordering). **Layer 3 is what's currently blocking BM25 reordering
in production**, by design per Reading 3 Pattern 2 retention.

**Reframing Reading 3's vindication.** Reading 3 is currently
load-bearing not because query-passing fails (it doesn't) and not
because tool semantics are wrong (they're not), but because **gating
discipline keeps BM25 default-off pending Stream D + dogfood +
production evidence**. Round 2 resolves Layers 1 and 2; Layer 3's
effect is the ship-state choice itself.

## Quality-shape asymmetry: audit vs architectural

**Round 1 (line-cited, source-grounded).** Independently verifiable
line-by-line via direct source-file inspection. Authoritative on
implementation specifics: exact field names, line numbers, function
signatures, SQL clauses. Suited to debugging-style or
implementation-detail questions.

**Round 2 (claim-cited, atlas-grounded).** Synthesized from extracted
claim text; surfaces cross-document architectural relationships that
source-file inspection would miss. **Q1 example: Round 2 surfaced
ADR-08 frontmatter broadcast â†’ ADR-09 name-overlap tiebreaker as
load-bearing architectural dependency.** That connection lives in
claim text (the "why" reasoning), not in source files. Round 1
reading `pipeline.ts` couldn't have found it.

**Different value props for different developer tasks.** For audit /
reproducibility purposes, Round 1's grounding is stronger. For
architectural-understanding purposes (the dogfood-tested shape),
Round 2's synthesis is more useful. **At N=1 this validates the
atlas-as-architectural-context design philosophy.** Both modes
correct; design intent (per CLAUDE.md "tool scope philosophy")
privileges architectural understanding over implementation lookup;
Round 2 delivers.

## Methodological observations

**(a) Live BM25-off observation as exceptional signal.** Round 2 Q3
demonstrated novel atlas observability via returned-content shape
inspection â€” Claude Code reasoned about server gating state from the
ordering of returned claims, not from any explicit metadata. This
under-explored signal channel suggests dogfood evidence can capture
gating-state effects without instrumentation.

**(b) Wall-clock trade-off (~2.2x slower per question).** Round 2 was
~78 sec/question vs Round 1 ~35 sec/question. MCP round-trip latency
(MCP protocol overhead + atlas DB query + response serialization)
exceeds direct OS file I/O. Real production-positioning concern;
developers experience latency directly.

**(c) Token-burn hypothesis (untested).** Round 1 read full files
(`pipeline.ts` ~717 LOC, `symbol-context.ts` full, etc.) â€” thousands
of tokens consumed. Round 2 used bundles which are MUCH smaller.
Token-efficiency value-proposition holds even with wall-clock
penalty; **should be measured at Stream D scale alongside latency**.
Round 2's slower wall-clock is genuine cost; the offsetting benefit
(token-burn reduction = the production-tool framing's central
value-prop) is plausible but currently unmeasured.

**(d) Parallel batching emerged naturally.** Round 2 Q3 used parallel
multi-symbol `get_symbol_context` + `find_by_intent` in a single
round-trip; Q1 also used parallel `find_by_intent` batches. MCP
design composes well with parallel tool calls without explicit
prompting.

**Priority: v0.3** (silent search miss; affects production query
patterns).

**1.4.7 â€” BM25 server flag default-off creates query-parameter
silent ignore.**
*Trigger:* Round 2 Q3 â€” Claude Code passed `query` parameter to

shape; less-attentive callers wouldn't notice.

flag status matters. Tool description engineering preferable to
error path since query-passing without BM25 may be desired in some
configs.
**Priority: v0.4** (depends on Step 7 ship default reconsideration;
if Fix 3 default-on lands at v0.4, this finding becomes moot).

## Scope limitation: explanation tasks vs code-change tasks

Round 1 + Round 2 tested architectural-understanding tasks ("explain
how X works"). Production Claude Code usage is heavily weighted
toward code-change tasks: implementing features, fixing bugs,
refactoring, debugging failing tests. The two task shapes have
different value-proposition tests:

**Explanation-task value proposition (this dogfood measured).** Does
atlas surface architectural understanding when discovered? Round 2
confirms: yes, with architecturally richer answers than file-reading.

**Code-change-task value proposition (untested).** Does atlas inform
change correctness? Specifically: does Claude Code consult MCP before
generating changes; does `impact_of_change` activate on edit-relevant
tasks; do generated changes respect architectural constraints
surfaced in the atlas; do constraint conflicts get flagged
proactively?

Notable: **`impact_of_change` had zero invocations across both
rounds.** Its specifically-code-change-task design wasn't exercised
by explanation questions. This is consistent with task-shape
mismatch (asking explanation questions doesn't trigger blast-radius
reasoning), not necessarily an `impact_of_change` activation gap.

**Implication for v0.4 dogfood scope.** Round 3 protocol designed
around code-change tasks would test the higher-stakes value
proposition. Suggested shape: cold or warmed session; ask Claude
Code to make small architecturally-constrained changes to
ContextAtlas (e.g., add config option respecting existing schema;
fix bug while preserving Pattern 2 semantics; refactor while
respecting ADR-15 multi-symbol uniformity). Observe MCP tool
activation, `impact_of_change` firing, whether generated changes
cite or implicitly respect surfaced constraints, whether constraint
conflicts get flagged proactively.

**Implication for Stream B planning.** Docstring extraction expands
Layer 3 content surface for repos without ADR culture. Stream B
value depends on whether expanded content informs code-change tasks
(currently unmeasured) just as much as explanation tasks (Round 2
measured positively). Stream D beta-ca measurement should consider
including code-change scenarios alongside explanation-question
scenarios in its measurement matrix â€” not as scope addition but as
scope clarification.

## Honest framing â€” what dogfood does and doesn't prove

Five caveats:

**(a) N=1 in every dimension.** One developer (Travis), one repo
(ContextAtlas itself), one session per round, three questions per
round, one Claude Code instance type, one workspace. Strong
directional signal; not statistical.

**(b) Question-shape narrowness.** Three questions all
explanation-task shape. Code-change task shape (production-dominant)
untested â€” see Subsection 10.5 for full scope-limitation treatment +
Round 3 protocol proposal.

**(c) BM25 flag off in test.** Layer 3 (server gating) wasn't
actually exercised. Reading 3's reordering-effectiveness question is
still empirically open; this dogfood tests Layers 1-2 only. Round 2
Q3 detected the off-state but didn't measure BM25-on effectiveness.

**(d) Stream D measurement still load-bearing for definitive
activation evidence.** This dogfood informs Step 8 step-planning +
v0.4 Fix 3 reconsideration; Stream D Step 14/15 measure activation
at scale across all three benchmark targets (per Step 7 progress log
Subsection 5(c) cross-severity tracked metric + grep-ceiling
closure validation).

**(e) Decision-under-uncertainty discipline preserved.** Reading 3
was correct given what was known at Step 7. Dogfood resolves
uncertainty for future decisions; doesn't validate Step 7
retrospectively. Per Phase 3 D5 synthesis Subsection 6 framing.
