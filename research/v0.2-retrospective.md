# v0.2 retrospective

**Status:** Brief retrospective on v0.2 execution. Authored
2026-04-25, day v0.2 shipped (tag `v0.2.0`, contextatlas commit
`fec0d74`). Input to v0.3 prep — what to keep doing, what to
adjust before the next version starts. Not a full post-mortem;
not a recap. Honest specifics over abstractions.

## Shipped vs. planned

v0.2 shipped 12 numbered steps (1–12) plus four sub-steps (4b,
4c, 11-infra, 11-prompts) over ~3 calendar days of execution.
All four v0.2-SCOPE.md success criteria satisfied via committed
artifacts. ADR-14 (gopls) the only new ADR. Atlas schema bumped
1.1 → 1.2 (additive `parent_id`). Three reference runs landed
(Phase 5 amendments, Phase 6 httpx fresh, Phase 7 cobra fresh).
Three v0.3+ findings logged from Phase 7. No descope; no rescope;
the original four success criteria all reached without
modification.

## What worked

### R2-atomic commit discipline held throughout

Every step's work decomposed into commits each of which is
independently sensible. The clearest exemplar was Step 7's
seven-commit arc (research note → harness fix → artifact
promotion → Phase 5 amendments → Phase 6 amendments → main-repo
progress log). Reviewers can read the arc in order; a `git
revert` of any one commit produces a coherent prior state. Step
9's six-commit arc (storage migration → types → skeleton →
listSymbols → findReferences → conformance) is the second
exemplar. The discipline survived even under scope expansion:
both Step 4 (Gaps 1–5) and Step 9 (parent_id prerequisite)
absorbed sub-steps without sacrificing per-commit atomicity.

### Pre-implementation surveys caught hidden work before it bit

Every step started with a survey — read existing code, read
ADRs, identify dependencies, propose approach, get sign-off, then
execute. This pattern caught Step 9's `parent_id`-not-on-Symbol
gap before any adapter code was written. It caught Step 11's
`extract-benchmark-atlas.mjs` cobra-whitelist gap (~5 LOC) before
matrix launch. It caught the `workspace/configuration` and `go on
PATH` gopls prerequisites during Step 8's probe iteration, before
ADR-14 was published. The cost of a 10-minute survey paid back
hours of mid-implementation debugging that didn't happen.

### ADR-13 → ADR-14 probe-then-decision pattern generalized cleanly

ADR-13 (Pyright) established the pattern: probe phase produces
empirical findings doc; ADR ratifies decisions citing those
findings; adapter implementation references the ADR. ADR-14
(gopls) followed the same shape with no friction. The Step 8
probe surfaced two non-obvious gopls runtime prerequisites
(PATH-resolved `go`, length-matched `workspace/configuration`)
that were locked in ADR-14 before adapter code shipped. By the
time GoAdapter implementation started in Step 9, every
structural decision was already settled — adapter LOC went into
mapping, not into rediscovering decisions.

### Phase 6 + Phase 7 synthesis depth surfaced findings beyond replication

Both reference runs went past "thesis replicates" into specific
mechanism findings. Phase 6 §5.1 found the p4-stream-lifecycle
claim-attribution + ranking precision issue (an alpha-vs-ca
finding that didn't depend on a CA-tools-broken interpretation).
Phase 7 §5.1/§5.2/§5.3 logged three v0.3+ findings: Go
grep-ability paradigm sensitivity, atlas-file-visibility
benchmark methodology issue, cross-harness asymmetry hypothesis.
None of these would have surfaced in a "did the matrix run
clean? yes? ship" pattern. The investment in synthesis depth was
v0.2's most concentrated source of v0.3 deliverables.

### Verification discipline prevented at least three published-but-wrong claims

Three v0.2 cycle moments where verification (or lack of it) was
load-bearing:

1. **Step 7 / Phase 5 §4.3 correction.** Original §4.3
   hypothesized "model mis-labels MCP responses as
   permission-denials." Verifying actual `result_preview`
   content showed 52/52 calls genuinely blocked at the CLI
   permission layer. Trusting the original framing would have
   missed the harness bug entirely.
2. **Step 9 / parent_id-not-on-Symbol catch.** ADR-14 specified
   `Symbol.parent_id`; verifying against the actual
   `src/types.ts` interface at survey time caught the gap before
   adapter code was written. Resulted in a clean Commit-1 storage
   migration v4 + atlas schema v1.2 prerequisite rather than a
   mid-implementation refactor.
3. **Step 11 / cobra c4 grep-ability framing.** c4 was a
   win-bucket cell where CA went net-negative on cost (+41% vs
   alpha). Default interpretations would have been "atlas gap"
   or "extraction quality issue." Reading the actual traces
   showed alpha used a single regex-OR Grep call to retrieve
   three symbols; CA fragmented into three `get_symbol_context`
   calls. The mechanism was Go's exported-symbol naming
   convention enabling regex disjunction, not an adapter
   problem. This shaped the v0.3 deliverable (multi-symbol call
   shape) correctly rather than triggering remediation work.

The discipline costs minutes per claim; the cost of an
unverified claim that reaches `main` is hours of correction
plus credibility hit. Documented as a feedback memory.

### Bucket framing held under losses

Four cells where the result was tempting to reframe but the
bucket-aware framing held — discipline started in Phase 6 (p4),
carried through to Phase 7 (c4 / c5 / c6). Two-phase survival,
not Phase-7-reactive:

- **c6 trick bucket (cobra Phase 7).** Apparent beta-ca "win"
  of −5 calls / −44% tokens. Investigation revealed it was a
  measurement artifact — beta tried to interrogate the visible
  `atlases/cobra/` directory instead of reading source. Phase 7
  §5.2 honestly excluded c6 from the beta-ca-vs-beta delta
  narrative rather than reporting it as a CA win. Filed as
  v0.3 backlog (atlas-file-visibility methodology).
- **c5 tie bucket (cobra Phase 7).** CA went net-negative on
  tokens (+22%). Reported as bucket-predicted ("c5 loses as
  predicted by bucket"), not relabeled as a CA quality issue or
  hidden from delta tables.
- **p4 win bucket (httpx Phase 6).** CA went net-negative on a
  *win-bucket* cell — surface-tempting framings were "atlas
  coverage gap" or "CA over-engineers." Phase 6 §5.1
  investigation surfaced claim-attribution + ranking precision
  as the real mechanism. Categorized as a v0.3+ CA limitation
  requiring work, not handwaved.
- **c4 win bucket (cobra Phase 7).** Mirror pattern to p4.
  Investigation surfaced Go's exported-symbol naming convention
  enabling regex disjunction (the grep-ability finding) rather
  than reframing as an adapter problem. Shaped a v0.3
  deliverable correctly. (Cross-reference: this is the third
  verification-discipline example below.)

The bucket framing is now empirically validated across three
target languages — h5/h6, p5/p6, c5/c6 all behave per RUBRIC.
Worth defending in v0.3 rather than relitigated.

### Cost ceilings were padding, not constraints

Phase 5 hono ran at $14.21 against $14 ceiling (over by 1.5%).
Phase 6 httpx ran at $8.11 against $18 ceiling (54% under).
Phase 7 cobra ran at $7.19 against $18 ceiling (60% under). The
v0.2-SCOPE.md "$13–16 per Go matrix" projection turned out to be
conservative against actual measured cost — useful padding for
budgeting but should be calibrated downward in v0.3 if smaller
ceilings would still leave headroom.

## What was painful

### Step 4 scope expansion (5 gaps from a planned 1–2)

Phase C hono spot-check at Step 4 surfaced 5 gaps in the TS
adapter when the scope assumed 1–2. Rescope discussion mid-step
to commit Gaps 1/2/3/5 (Gap 4 deferred to v0.3). Added Step 4b
(hono atlas re-extraction) and Step 4c (Phase 5 spot-check on
h4-ca to verify thesis under refined extraction). Total Step 4
arc grew from ~1 day to ~2 days. The pattern worked — bundle
sub-steps under originating step rather than splitting — but it
delayed Stream B's start. **Lesson:** parity-check phases need a
bigger upfront sample (Phase C did 3 hono files; should have
been 5–6) so gap surface is realistic before scope locks.

### Step 7 was the most consequential surprise — and the most foundational fix

Phase 5 §4.3 had stood unchallenged through Phase 6's writeup.
Step 7 investigation revealed it was wrong: 100% MCP block rate
across all 11 beta-ca cells, not "model mis-labeling." Required
re-running 11 cells, amending Phase 5 §4 + §4.3 + §7.2,
amending Phase 6 §4 + §5.3, preserving v1 artifacts as
`-v1-permission-blocked.json` for audit, adding a methodology
finding. Total session-time: ~3 hours of re-execution + ~3 hours
of doc amendments. The painful side: a permission-block sentinel
string was always visible in v1 traces — earlier verification
of cell artifacts during Phase 5 synthesis would have caught it
sooner.

**The recovery is what worked, and is the central inheritable
v0.2 lesson for v0.3.** Verification discipline (see "What
worked" §3 above) is the mechanism that surfaced the truth:
reading actual `result_preview` content rather than trusting
the prior framing. The MCP preflight guard added in Step 11
caught a real regression during the self-test (when
`--allowedTools` was temporarily removed to verify guard
behavior, preflight aborted with the expected named-flag
error). Without Step 7's recovery, Phase 6/7 would have
inherited the silent-invalidation pattern — the cobra
reference run would have shipped with the same buried defect
hono and httpx had carried for weeks.

**For future readers asking "what should v0.3 inherit from
v0.2?":** Step 7's outcome is the central example. The
verification-discipline + methodology-improvement-commit
pattern (research note → harness fix → preflight guard) is
the mechanism that turns a painful discovery into durable
harness hardening. Apply it whenever a "weird but documented"
pattern surfaces and nothing depends on the documented
explanation being right.

### Step 9 LOC trajectory was wrong twice

Pre-implementation estimate: ~700 LOC across 5 commits. Actual:
~996 LOC across 6 commits (parent_id storage prereq absorbed
mid-step). Mid-implementation estimate after Commit 3: trending
toward 1250 LOC; potential ADR-14 budget violation flagged.
Final after Commits 4–5: 996 LOC, exactly at ADR-14's upper
bound. Two estimate corrections required mid-arc; both turned
out to be over-corrections. **Lesson:** LOC estimates for
adapter-skeleton phases (initialize lifecycle + LSP wire types +
helpers) systematically undercount. Future budget framing should
allocate more upfront to the skeleton and trust that data-method
phases come in lean if the probe + skeleton work is solid.

### Synthesis-note LOC estimates were finding-blind

STEP-PLAN-V0.2.md Step 11 specified "~200–300 LOC" for the
Phase 7 synthesis; actual was 540 LOC. Phase 5 (848 LOC
post-amendments) and Phase 6 (593 LOC) had the same shape
against their initial estimates. **The synthesis lengths
reflect content depth, not padding** — each per-finding §5.x
section runs 30–60 LOC when it carries a v0.3+ implication,
executive summaries need depth to make findings citable, and
cross-language comparison sections grow as comparison
substrate grows. The estimate was the problem, not the output.
Concrete adjustment captured under "What to adjust for v0.3"
below — synthesis budgets are finding-scaled, not
replication-scaled.

### Step 11 sequencing required explicit prep

Step 11's gating sequence — preflight infra commit → prompts
pre-registration commit → atlas extraction → matrix run →
artifact promotion → synthesis — was non-obvious. The first
attempt to start Step 11 conflated atlas extraction with prompt
authoring. Explicit pre-session sequencing pause + dictation of
the 7-step order resolved it, but cost ~30 min of conversation
time before any code shipped. **Lesson:** later-in-stream steps
that compose multiple subtasks need a sequencing diagram or
explicit ordering rule baked into the step body, not deferred
to execution-time discovery.

## Minor observations

Behavioral observations from v0.2 execution that are worth
noting for v0.3 prep but don't rise to the level of pain points
or process changes.

- **MCP preflight inconclusive-path occurred 1 of 3 times in
  self-test.** The explicit-instruction probe prompt ("Use the
  mcp__contextatlas__find_by_intent tool exactly once...") works
  most of the time but isn't 100% deterministic — one self-test
  run returned inconclusive (no MCP tool call attempted). The
  inconclusive-path warning handled it correctly. Worth tracking
  through v0.3 to see if a more forcing probe shape is feasible
  (or accept inconclusive as warning-level rather than zero-noise).

## What's worth carrying into v0.3

1. **Pre-implementation survey before every step.** Time-bounded
   (10–20 min). Output is a written proposal: scope per file,
   commit splits, dependencies, open questions. Survey caught
   substantial work in v0.2; same pattern should hold in v0.3.

2. **R2-atomic commits with explicit progress-log entries on
   ship.** Per-step progress log entries in
   `STEP-PLAN-V0.3.md` (or equivalent) preserve decisions that
   would otherwise rot. v0.2's progress logs are already showing
   value as references in v0.3 prep.

3. **Verification discipline on every research-note claim.**
   Documented as a feedback memory. Three v0.2 grounding
   examples make the case concrete for future-self.

4. **Bucket-aware methodology with three-language baseline as
   defended starting point.** The c1/h1/p1 invariant is a real
   finding; tie/trick losses are by-design. v0.3 reference runs
   should preserve the framing rather than re-litigate it.

5. **ADR-N → probe-findings → decision block pattern.**
   ADR-13/14 demonstrate it works for adapter-class decisions.
   v0.3's claim-source-enrichment work likely needs an ADR for
   the docstring/README/PR ingestion shape — the probe-first
   approach should apply.

## What to adjust for v0.3

1. **LOC estimates: pad skeleton phases, trust data-method
   phases to come in lean.** Explicit upfront allocation
   ratio: 50–60% to skeleton + infrastructure, 30–40% to
   data methods, 5–10% to wiring/conformance. Calibrated against
   v0.2 actuals.

2. **Synthesis-note LOC budget should scale with finding
   count.** Default: 400–550 LOC for a 3-findings synthesis.
   Don't pre-anchor on 200–300 unless the run is pure
   replication with no investigation findings.

3. **Parity-check / probe phases need bigger upfront samples.**
   v0.2 Step 4's 3-file Phase C undercounted gap surface;
   future similar phases should sample 5–8 files before scope
   locks.

4. **Sequencing rules baked into composite-step bodies.**
   Step 11's preflight → prompts → atlas → matrix → promotion
   → synthesis ordering should have been a numbered sub-list in
   the step body, not discovered at execution time. Apply same
   rule to v0.3 composite steps.

5. **Methodology-issue research-notes filed during execution,
   not at retro time.** v0.2 filed
   `atlas-file-visibility-benchmark-methodology.md` as a v0.3
   backlog candidate but the note itself wasn't authored. v0.3
   should author methodology-issue notes the same session they
   surface, while context is loaded.

6. **MCP preflight probe shape iteration.** Track inconclusive
   rate across v0.3 reference runs; if the rate stays >0,
   investigate forcing probe prompts (or accept inconclusive as
   warning-level rather than zero-noise).

## What v0.3 will look like (preview, not commitment)

v0.3 scope per ROADMAP.md combines original-scope claim source
enrichment (docstring extraction, README/CONTRIBUTING.md
parsing, git commit-message extraction beyond the v0.1 regex,
PR description mining) with three v0.2 follow-throughs (multi-
symbol `get_symbol_context` call shape, atlas-file-visibility
methodology fix, cross-harness asymmetry tracking). Both work
streams compose. v0.3 is the largest version on the roadmap so
far in terms of accumulated scope. Step-plan authoring should
expect 12–18 numbered steps depending on how the streams
decompose.

The retro doesn't author v0.3-SCOPE.md. That's the next session
after the backlog walk.
