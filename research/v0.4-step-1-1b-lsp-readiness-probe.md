**`$/progress` notifications received:** YES (2 frames; one

| Event | Δt from didOpen | Payload |
|---|---|---|
| `window/logMessage` (typescript version banner) | +0ms | "Using Typescript version (workspace) 5.9.3 from path …" |
| `$/typescriptVersion` (tsserver-specific) | +2ms | — |
| `window/workDoneProgress/create` (server→client) | +182ms | `token=<uuid>` |
| `$/progress` BEGIN | +183ms | `kind=begin`, `title="Initializing JS/TS language features…"` |
| `$/progress` END | +1158ms | `kind=end` (no message) |

**Token shape:** Random UUID per session
(`d5162f1d-f24a-4c36-80ec-207ed04c6257` in this probe). Token is
session-scoped, not per-request.

**Title shape:** Hard-coded `"Initializing JS/TS language
features…"` (Unicode horizontal-ellipsis). Stable across sessions
per typescript-language-server source.

**Critical timing:** `$/progress` END fires **~84ms BEFORE** the
first `publishDiagnostics`. This is a usable readiness signal —
the END frame indicates tsserver finished its cold-start project-
load pass and is now able to push diagnostics.

**Scope of signal:** ONE-SHOT per session. The `$/progress`
sequence fires only during cold-start init; subsequent `didOpen`
notifications for additional files do NOT trigger new
`$/progress` flows. Therefore `$/progress` is a server-readiness
signal, **not** a per-file diagnostic-readiness signal.

## Cross-server synthesis

### Signal taxonomy reframe

The probe surfaces a distinction the kickoff-prompt design proposal
collapsed: there are TWO readiness signals at play, not one.

1. **Server-readiness signal (cold-start, one-shot per session).**
   "Server has finished init and is ready to compute diagnostics
   for any file." Emitted as `$/progress` END for tsserver + gopls;
   pyright has no equivalent surface and signals readiness only
   via the first `publishDiagnostics` push itself.
2. **Per-file readiness signal (per-call).** "Diagnostics for this
   specific URI have been computed." This IS `publishDiagnostics`
   for all three servers — there is no stronger per-file signal.

The v0.3 Commit 1.5 flake mode (`04f5b9d`) was a server-readiness
problem, not a per-file problem. tsserver cold-start ate the 1s
ceiling because the per-file `publishDiagnostics` push couldn't
happen until the ~1.2s cold-start finished. With server-readiness
absorbed at `initialize()` time via `$/progress` END, the per-call
ceiling can shrink toward "real per-file analysis time" rather
than "cold-start absorption budget."

### Decision-tree resolution

Tying the probe back to the decision tree:

- **3/3 clean signals:** falsified — pyright 1.1.409 emits no
  `$/progress` for cold-start.
- **2/3 clean signals:** matched — tsserver + gopls emit
  `$/progress` END before first `publishDiagnostics`; pyright does
  not.
- **1/3 clean signal:** falsified.
- **0/3 clean signals:** falsified.

**Decision: Option A with pyright-fallback** (kickoff prompt's
naming was Option-A-with-TS-fallback; probe inverts that — TS has
the cleanest signal, pyright is the fallback).

### Implementation shape (preview for Step 1.2-1.4)

The unified pattern that absorbs the probe findings:

1. **`LspClient` gains `onProgress(handler)` API.** Single dispatch
   point for `$/progress` notifications keyed by token. Lets each
   adapter register a handler that resolves a per-session
   "server-ready" promise on END frame for the cold-start token.
2. **Each adapter's `initialize()` awaits a server-ready promise
   before returning.** Race: `$/progress` END for the recognized
   cold-start token (TS title match: "Initializing JS/TS language
   features…"; gopls title match: "Setting up workspace") OR a
   per-adapter cold-start ceiling (TS: 5s; gopls: 5s; pyright:
   2s — pyright doesn't emit, so the ceiling fires every time but
   actual cold-start is ~300ms so the ceiling is generous).
3. **`getDiagnostics` keeps the publishDiagnostics-listener
   pattern but with tightened per-call ceilings.** With cold-start
   absorbed in `initialize()`, the per-call ceiling no longer
   needs to absorb cold-start variance:
   - TS: 5s → 1.5s (the v0.3 raise was specifically for
     cold-start; warm pushes are fast)
   - Pyright: 1s → 1s (unchanged; pyright cold-start absorbed by
     init-time ceiling and warm pushes already fit 1s)
   - Go: 2s → 1.5s (warm pushes; observed ~600ms cold)
4. **Pattern lift.** The shared listener+ceiling-race body lives
   in a helper (likely `src/adapters/diagnostics-readiness.ts`)
   imported by all three adapters.

### LOC estimate revised

| Component | Est. LOC |
|---|---|
| `LspClient.onProgress(handler)` + dispatch plumbing | ~30 |
| Shared `waitForDiagnostics` helper | ~40 |
| TS adapter init-time `serverReady` race + ceiling tightening | ~30 |
| Pyright adapter init-time race (no-progress fallback) + helper migration | ~20 |
| Go adapter init-time `serverReady` race + ceiling tightening | ~30 |

**Total ~150 LOC** — at the rescope-condition edge but achievable.
Pyright's no-progress fallback is the smallest delta because the
existing pattern is already correct for it; the change is just
helper migration.

### Open items the probe did NOT resolve

- **Per-file `publishDiagnostics` re-pushes (gopls's two-push
  behavior).** Out of scope for B2; relevant to v0.5+ if cache
  staleness becomes a quality-axis concern.
- **Pyright `$/progress` on large repos.** Probe used 5-file
  fixture; httpx (23 files) and larger may exhibit different
  pyright surface. Step 5 re-extraction provides incidental
  evidence; if pyright surfaces `$/progress` on real repos, the
  pyright path can opt in to the same pattern as TS + Go later
  without changing the v0.4 design.
- **Title-match brittleness.** TS "Initializing JS/TS language
  features…" and gopls "Setting up workspace" are hard-coded
  strings in their respective servers. Title-match works today;
  upstream change would silently break server-readiness detection
  and fall back to ceiling. Acceptable risk per ADR-discipline
  ("ceiling acts as safety net") — flag in Step 1.5 ADR
  amendments as a known limitation.

## Recommendation

**Resolve OQ4 as Option A with pyright-fallback.** Implementation
proceeds as Steps 1.2 (TS) → 1.3 (Pyright) → 1.4 (Go). The
substantive upgrade (server-readiness `$/progress` integration
absorbing cold-start) ships for tsserver + gopls; pyright keeps
the publishDiagnostics-first-push pattern with its tighter ceiling
and inherits the helper-lift mechanical cleanup.

The pattern stays unified at the helper level; per-adapter
configuration toggles `$/progress` opt-in. v0.5+ pyright migration
to `$/progress` (if upstream begins emitting) requires only
changing the pyright adapter's config — no helper change.

**Estimated total v0.4 Step 1 LOC: ~150**, within the rescope-
condition budget but on the edge. Step 1.2 implementation is the
canary — if TS adapter migration alone exceeds ~50 LOC, that's the
trigger to revisit the abstraction shape before propagating to
Steps 1.3 + 1.4.
